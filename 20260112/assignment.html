1 & 2. API Endpoints and HTTP Methods

The following table describes the CRUD operations for the Task resource:

Operation HTTP Method Endpoint Reasoning
Create Task POST /tasks Used to submit new data to the server. Non-idempotent.
Read All Tasks GET /tasks Used to retrieve a list of all resources. Safe method.
Read Specific Task GET /tasks/:id Used to fetch a single resource using its unique identifier.
Update Task PUT /tasks/:id Used to replace an existing resource with new data. Idempotent.
Delete Task DELETE /tasks/:id Used to remove a specific resource from the collection.

3. Unique Identification

In the absence of a database, the server will use a Global Counter Variable. Each time a POST request is made, the server assigns the current counter value to the new task and then increments the counter (e.g., ID 1, 2, 3...). This ensures no two tasks share the same ID during the application's runtime.

4. Data Volatility

Since data is stored in-memory (RAM), all data will be lost when the server restarts. This is because RAM is volatile. In a production DevOps environment, we would solve this by connecting the API to a persistent database or a mounted volume.

5. HTTP Status Codes

  • 201 Created: Returned after a successful POST request.
  • 200 OK: Returned for successful GET, PUT, or DELETE requests.
  • 400 Bad Request: Returned if the client sends invalid task data.
  • 404 Not Found: Returned if the client tries to access an ID that does not exist.
  • 500 Internal Server Error: Returned if the server encounters an unexpected crash.
